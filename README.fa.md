RainScanner آی‌پی‌های نسخه‌ی ۴ (IPv4) لبه‌ی CDNها را پیدا می‌کند که واقعاً به‌عنوان
ترنسپورت Xray کار می‌کنند و آن‌ها را بر اساس **تأخیر واقعیِ پروکسی‌شده** (نه پینگ
ICMP) مرتب می‌کند. دو مرحله دارد:

۱. **پیش‌فیلتر TCP** — عملیات اتصال همزمان بالا روی پورت(های) هدف، IPهای از کار افتاده را حذف می‌کند.

۲. **تأیید با Xray** — برای هر بازمانده، یک کانفیگ Xray ساخته می‌شود که مستقیماً به
آی‌پی کاندید وصل می‌شود و در همان حال SNI / WS-Host اصلی را حفظ می‌کند، سپس
درخواست‌های واقعیِ HTTP را از مسیر پروکسی می‌فرستد و تأخیر سرتاسری را می‌سنجد. فقط
آی‌پی‌هایی که K از N آزمون را در محدوده‌ی تأخیر مجاز پاس کنند پذیرفته می‌شوند.

فقط IPv4؛ بازه‌های IPv6 کنار گذاشته می‌شوند.

## 🌐 CDNهای پشتیبانی‌شده

هفت CDN پیش‌فرض: `cloudflare`، `fastly`، `cloudfront` (آمازون کلودفرانت)،
`arvan` (ابرآروان)، `Gcore`، `railway` و `vercel`.

- چهار مورد نخست، محدوده‌های رسمیِ منتشرشده‌ی خود را خودکار می‌گیرند و در
  `ips/<cdn>.json` کش می‌کنند. `railway` و `vercel` API عمومی برای محدوده‌ها ندارند،
  پس محدوده‌هایشان همراه مخزن ارائه و به‌صورت دستی نگه‌داری می‌شود.
- **ویرایش هر CDN پیش‌فرض** (محدوده‌ها یا API) یا **حذف** موارد بلااستفاده، مستقیم در
  محیط گرافیکی. یک CDN پیش‌فرضِ حذف‌شده در *reload all* بعدی برمی‌گردد.
- **افزودن CDN دلخواه** از محیط گرافیکی (**Add custom +**) — یک نام به‌علاوه‌ی فهرستی
  از CIDRها یا آی‌پی‌ها، در صورت تمایل همراه یک API برای به‌روزرسانی. موارد سفارشی در
  `ips/custom.json` می‌مانند و با reload بازنویسی نمی‌شوند.
- **کلید منبع (`github` ↔ `official api`).** گرفتن محدوده‌ها از آینه‌ی گیت‌هاب یا از
  API خود هر CDN — اگر API یک CDN در منطقه‌ی شما مسدود است، روی `github` بگذارید.
- مجموعه‌ی پیش‌فرض با یک **مانیفست** (`inside-api/index.json`) تعریف می‌شود، نه کدِ
  ثابت. یک GitHub Action روزانه محدوده‌های آینه‌شده را تازه می‌کند؛ برای افزودن CDN
  پیش‌فرض جدید، آن را به مانیفست اضافه کنید.

## 🔌 ترنسپورت‌ها

VLESS، VMess و Trojan روی `tcp` (به‌همراه HTTP obfs)، `ws`، `grpc` (به‌همراه حالت
`multi`)، `http/h2` و **`xhttp`** (به‌همراه `mode` و `extra` مانند
`{"xPaddingBytes":"100-1000"}`)، با `tls` یا `reality`. همه‌ی قابلیت‌های TLS روی
آزمون حفظ می‌شود تا تأیید با یک کلاینت واقعی هم‌خوان باشد: `fingerprint` (uTLS)،
`alpn`، `allowInsecure`، `flow` و **ECH** (`echConfigList`). آی‌پی کاندید در اوت‌باند
تزریق می‌شود و در همان حال SNI / Host اصلی حفظ می‌گردد تا لبه‌ی CDN درست مسیریابی کند.

## 🚀 اجرای برنامه

برنامه یک فایل اجرایی مستقل است؛ چیزی برای نصب نیست (xray داخل خود فایل قرار دارد).

```bash
# ویندوز: کافی است روی فایل دوبار کلیک کنید، یا:
rainscan.exe -serve

# لینوکس:
./rainscan-linux-amd64 -serve
```

سپس مرورگر روی این آدرس باز می‌شود:

```
http://127.0.0.1:8787
```

```bash
# اجرا داخل WSL و دسترسی از مرورگرِ ویندوز — همه‌ی واسط‌ها را bind کنید:
./rainscan.exe -serve -addr 0.0.0.0:8787
```

> مرحله‌ی ۱ (اسکن TCP) به هیچ چیز دیگری نیاز ندارد. فقط برای **مرحله‌ی ۲** (تأیید
> تأخیر واقعی) به یک لینک کانفیگ Xray نیاز دارید.

## 🧭 گام‌به‌گام: نخستین اسکن شما

**۱) یک CDN انتخاب کنید.** از نوار کناری (بخش **Targets**) روی یکی از CDNها مثل
`cloudflare` کلیک کنید. محدوده‌های آن در پنل **Selected Ranges** نمایش داده می‌شود. در
محیط گرافیکی، در هر اجرا فقط **یک CDN** اسکن می‌شود.

**۲) (اختیاری) منبع محدوده‌ها را انتخاب کنید.** اگر API رسمیِ یک CDN در منطقه‌ی شما
مسدود است، کلید **source** را روی `github` بگذارید تا محدوده‌ها از آینه‌ی گیت‌هاب خوانده
شوند؛ در غیر این‌صورت `official api` کافی است.

**۳) (اختیاری) یک لینک کانفیگ بدهید.** اگر می‌خواهید آی‌پی‌ها با **تأخیر واقعی** تأیید
شوند، یک لینک اشتراکی (`vless://` یا `vmess://`) را در کادر **Optional config**
بچسبانید. اگر خالی بگذارید، اسکن در حالت **فقط TCP** انجام می‌شود (فقط در‌دسترس‌بودن،
بدون سنجش تأخیر).

**۴) (اختیاری) تنظیمات پیشرفته.** بخش **⚙ Advanced settings** را باز کنید تا پورت‌ها،
موازی‌سازی و معیارهای تأیید را تنظیم کنید (در ادامه توضیح داده شده). برای شروع، مقادیر
پیش‌فرض مناسب‌اند.

**۵) اسکن را شروع کنید.** روی **Start scan** بزنید. در حین اسکن، کارت‌های داشبورد،
**Live Log** و نوار پیشرفت (با ETA) زنده به‌روز می‌شوند. برای توقف، همان دکمه که اکنون
**Stop** شده را بزنید.

**۶) نتیجه را بردارید.** پس از پایان اسکن:
- **Valid IPs → TCP reachable** — آی‌پی‌هایی که پورت‌شان باز بود.
- **Xray confirmed** — آی‌پی‌هایی که تأخیر واقعی‌شان تأیید شد (اگر لینک داده باشید).
- **Output configs** — همان لینک کانفیگ شما، اما با آی‌پی تمیزِ جایگزین‌شده، آماده‌ی
  استفاده. با دکمه‌ی **copy** آن را بردارید.

## 🖥️ محیط گرافیکی، بخش به بخش

### نوار بالا (Header)
لوگو، نام و نسخه‌ی برنامه (`v2.0.0`) و یک پیوند به مخزن **GitHub**. روی صفحه‌های کوچک،
دکمه‌ی منو (☰) نوار کناری را باز می‌کند.

### نوار کناری — بخش Targets
- **کلید source (`github` / `official api`)** — تعیین می‌کند محدوده‌ها از کجا گرفته
  شوند: آینه‌ی گیت‌هاب یا API رسمیِ هر CDN.
- **دکمه‌ی `⟳` (reload all)** — محدوده‌های همه‌ی CDNها را یکجا دوباره می‌گیرد. چون
  مرجع CDNهای پیش‌فرض گیت‌هاب است، هر CDN پیش‌فرضی را که حذف کرده‌اید **برمی‌گرداند** و
  محدوده‌هایش را تازه می‌کند — اما CDNهای سفارشی شما دست‌نخورده می‌مانند.
- **فهرست CDNها** — هر مورد تعداد محدوده‌هایش (`N range(s)`) و یک مداد ✎ برای
  **ویرایش** دارد. روی نام کلیک کنید تا انتخاب شود.
- **`+ Add custom`** — افزودن یک هدف سفارشی: یک نام به‌علاوه‌ی فهرستی از CIDRها یا
  آی‌پی‌ها (و در صورت تمایل یک API برای به‌روزرسانی).
- **`⚙ Advanced settings`** — تنظیمات پیشرفته (پایین توضیح داده شده).
- **`Start scan`** — شروع/توقف اسکن.

### تنظیمات پیشرفته (Advanced settings)
- **Network**
  - **Ports to scan** — پورت‌ها برای اسکن، با کاما جدا می‌شوند (مثلاً `443,80`). هر
    آی‌پی روی همه‌ی این پورت‌ها بررسی می‌شود. پیش‌فرض `443`.
  - **Xray binary path** — مسیر فایل xray (خالی = تشخیص خودکار).
- **Performance**
  - **Lite mode** — حالت کم‌مصرف؛ موازی‌سازی را برای رایانه‌های ضعیف محدود می‌کند.
  - **TCP concurrency / Xray procs / Batch size** — میزان موازی‌سازی (خالی = خودکار،
    متناسب با CPU).
- **Stage 2 · Xray confirm**
  - **Probes / Confirm** — از هر N درخواست واقعی، چند موفقیت لازم است (مثلاً ۳ از ۵).
  - **Max latency (ms)** — سقف تأخیر مجاز (میانه).
  - **Probe timeout (ms)** — مهلت هر درخواست (خالی = خودکار).
- **Sampling**
  - **Random sample** — حداکثر تعداد آی‌پیِ تصادفی برای اسکن (`0` = همه).
  - **Sample per /24 / Max hosts/CIDR** — محدودکردن تعداد آی‌پی از هر بازه.
  - **Force re-fetch** — گرفتن دوباره‌ی محدوده‌ها از منبع پیش از اسکن.

### داشبورد و نتایج (بخش اصلی)
- **کارت‌های آماری** — زنده: **Hosts Scanned** (آی‌پی‌های بررسی‌شده)، **TCP Open**
  (پورت باز)، **Confirmed IPs** (تأییدشده با Xray)، **Ranges Loaded** (تعداد محدوده‌ها).
- **نوار پیشرفت** — درصد، زمان سپری‌شده و زمان تخمینی باقی‌مانده (ETA).
- **Selected Ranges** — پیش‌نمایش محدوده‌های CDN انتخاب‌شده.
- **Live Log** — گزارش زنده‌ی رویدادهای اسکن.
- **Valid IPs** — آی‌پی‌های سالم؛ بخش **TCP reachable** و بخش **Xray confirmed**،
  هرکدام با دکمه‌ی **copy IPs**.
- **Optional config** — کادر چسباندن لینک اشتراکی (برای تأیید تأخیر واقعی).
- **Output configs (clean IP inside)** — لینک کانفیگ شما با آی‌پی تمیزِ جایگزین‌شده،
  آماده‌ی استفاده؛ با **copy all** کپی می‌شود.

### پنجره‌ی ویرایش/افزودن CDN
با کلیک روی مداد ✎ یا **Add custom** باز می‌شود:
- **name** — نام هدف (برای CDNهای پیش‌فرض قابل تغییر نیست).
- **api url** — نشانی برای گرفتن خودکار محدوده‌ها (اختیاری). با **⟳ Reload range IPs**
  می‌توانید همان لحظه محدوده‌ها را از آن بگیرید.
- **ip ranges** — فهرست CIDRها/آی‌پی‌ها، هر کدام در یک خط.
- **Save / Cancel / Delete** — ذخیره، انصراف، یا حذف هدف.

## 📦 نصب و راه‌اندازی

RainScanner روی **ویندوز** و **لینوکس** اجرا می‌شود (آزمایش‌شده روی
**WSL Ubuntu 24.04**).

### استفاده از نسخه‌ی منتشرشده (پیشنهادی)

فایل اجرایی متناسب با سیستم‌عاملتان را از صفحه‌ی
[Releases](https://github.com/IzumiRain/RainScanner/releases) دانلود و اجرا کنید —
**xray-core داخل خود فایل قرار دارد**، پس چیز دیگری برای نصب نیست. در نخستین اجرا به
یک پوشه‌ی کش کاربری استخراج می‌شود.

```powershell
.\rainscan-windows-amd64.exe          # محیط گرافیکی روی http://127.0.0.1:8787 (با دوبار کلیک هم باز می‌شود)
```

```bash
./rainscan-linux-amd64 -serve         # محیط گرافیکی روی http://127.0.0.1:8787
# از مرورگر ویندوز به یک نمونه‌ی WSL، همه‌ی واسط‌ها را bind کنید:
./rainscan-linux-amd64 -serve -addr 0.0.0.0:8787
```

### ساخت از سورس

به [Go 1.26+](https://go.dev/dl/) نیاز دارید. یک `go build` ساده یک جای‌گیرنده‌ی خالی
برای xray جاسازی می‌کند؛ برای جاسازیِ xray واقعی ابتدا `scripts/fetch-xray.sh` را اجرا
کنید (همان کاری که نسخه‌ی منتشرشده می‌کند)، **یا** کافی است یک فایل `xray` / `xray.exe`
روی `PATH` داشته باشید (یا با `--xray /path/to/xray` بدهید) و این مرحله را رد کنید.

```bash
git clone https://github.com/IzumiRain/RainScanner.git && cd RainScanner
./scripts/fetch-xray.sh               # اختیاری: جاسازی xray در فایل اجرایی
go build -o rainscan ./cmd/cdnscan    # روی ویندوز rainscan.exe
./rainscan -serve
```

> مرحله‌ی ۱ (اسکن TCP) بدون هیچ xrayای کار می‌کند؛ xray فقط برای تأیید تأخیر واقعیِ
> مرحله‌ی ۲ لازم است. کانفیگ‌های آزمون از مسیریابیِ geo استفاده نمی‌کنند، پس
> `geoip.dat` / `geosite.dat` **لازم نیست**.

## ⌨️ استفاده از خط فرمان

محیط گرافیکی یک CDN را اسکن می‌کند؛ خط فرمان می‌تواند چند CDN را اسکن کند (`-cdn a,b`).

```bash
# فهرست ارائه‌دهنده‌ها
./rainscan.exe -list

# اسکن کامل با تأیید تأخیر Xray
./rainscan.exe \
  --cdn cloudflare,fastly \
  --link "vless://UUID@domain:443?security=tls&type=ws&host=cdn.domain&path=%2Fws&sni=cdn.domain" \
  --port 443 \
  --tcp-concurrency 1000 \
  --xray-concurrency 32 \
  --probes 5 --confirm 3 --max-latency 800ms

# بازه‌های بزرگ: به‌جای شمارش کامل، نمونه‌گیری کنید
./rainscan.exe --cdn cloudflare --link "vmess://..." --sample-per-24 4

# حالت فقط‑TCP (بدون لینک) — فقط در‌دسترس‌بودن، بدون تأیید تأخیر
./rainscan.exe --cdn fastly --sample-per-24 1
```

محدوده‌های کش‌شده در `ips/<cdn>.json` و نتایج تأییدشده در `results/<cdn>.json` ذخیره
می‌شوند.

> **نکته:** فایل `results/<cdn>.json` آی‌پی‌های تمیزِ رتبه‌بندی‌شده را نگه می‌دارد
> (`ip`، `port`، `median_ms`، `successes`، `total`) — **نه** لینک‌های کامل را. لینک‌های
> آماده با آی‌پی تمیزِ جایگزین‌شده در محیط گرافیکی ساخته و با دکمه‌های copy گرفته
> می‌شوند؛ در آن فایل JSON نوشته نمی‌شوند.

## ⚡ سرعت و پیشرفت

- **آزمونِ شکست سریع:** مرحله‌ی ۲ به‌محض مشخص‌شدن نتیجه، کاندید را متوقف می‌کند — به‌محض
  رسیدن به تعداد موفقیتِ `--confirm` می‌پذیرد و به‌محض غیرممکن‌شدنِ آن، رد می‌کند.
  آی‌پی‌های در‌دسترس‌اما‌مرده دیگر کل بودجه‌ی آزمون را مصرف نمی‌کنند (در عمل حدود ۳٫۵
  برابر سریع‌تر).
- **نوار پیشرفت زنده + ETA** هم در CLI (نوار درجا با زمان سپری‌شده/تخمینی) و هم در محیط
  گرافیکی. برای موازی‌سازی بیشتر، `--xray-concurrency` را بالا ببرید.

## 🎯 کنترل‌های دقت

- `--probes` / `--confirm` — نیازمند K موفقیت از N درخواستِ واقعیِ پروکسی‌شده.
- یک درخواست گرم‌کردنِ دورانداخته، هزینه‌ی برپایی تونل را از میانه حذف می‌کند.
- تأخیرِ میانه (نه میانگین) در برابر نوسان مقاوم است؛ `--max-latency` بودجه است.
- پذیرش نیازمند موفقیتِ واقعیِ پروکسی‌شده است (مثبتِ کاذب را حذف می‌کند)؛ تکرار در چند
  آزمون، منفیِ کاذبِ گذرا را سرکوب می‌کند.

## 🚩 پرچم‌های مهم

| پرچم | پیش‌فرض | معنی |
|------|---------|------|
| `--cdn` | — | CDNهایی که اسکن می‌شوند (`a,b` یا `all`) |
| `--link` | — | لینک اشتراکیِ `vless://` / `vmess://`؛ برای فقط‑TCP خالی بگذارید |
| `--xray` | PATH | مسیر فایل اجرایی xray-core |
| `--port` | 443 | پورت هدف TCP |
| `--ports` | — | فهرست پورت‌ها با کاما، مثل `443,80,8080` (جایگزین `--port`) |
| `--tcp-concurrency` | 1000 | تعداد اتصال‌های موازیِ مرحله‌ی ۱ |
| `--xray-concurrency` | 32 | تعداد پردازه‌های موازیِ xray در مرحله‌ی ۲ |
| `--probes` / `--confirm` | 5 / 3 | تأیید K از N |
| `--max-latency` | 800ms | بودجه‌ی تأخیر (میانه) |
| `--sample-per-24` | 0 | نمونه‌گیری N میزبان از هر /24 (۰ = کامل) |
| `--refresh` | false | گرفتن دوباره‌ی اجباریِ محدوده‌های ارائه‌دهنده |

دکمه‌ی تک‌کلیکیِ `⟳ reload all` (بازگرداندن CDNهای پیش‌فرضِ حذف‌شده، با حفظ موارد
سفارشی)، CDNهای مبتنی بر مانیفست با به‌روزرسانی روزانه، دو CDN پیش‌فرضِ جدید
(`railway`، `vercel`)، اسکن چندپورتی، و داشبورد بازطراحی‌شده.

## 🤝 مشارکت

از Issue و Pull Request استقبال می‌شود. اگر باگی دیدید، به CDNای برخوردید که
محدوده‌هایش نادرست به‌نظر می‌رسد، یا ایده‌ای دارید،
[یک Issue باز کنید](https://github.com/IzumiRain/RainScanner/issues) یا یک
[Pull Request](https://github.com/IzumiRain/RainScanner/pulls) بفرستید — افزودن یک
CDN پیش‌فرضِ جدید می‌تواند به کوچکیِ یک ورودی در مانیفست باشد.
